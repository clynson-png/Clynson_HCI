# SPEC: Reports A4 Premium, Athlete Portal, and Premium Access Gate

Date: 2026-06-28

Project root: `C:\HCI_REACT_WEB\REACT_WEB_TAURUS`

Status: product and implementation spec

## Objective

Review the output format of all reports generated by the React app and create a responsive React athlete portal with login. The athlete must be able to access the TAURUS, Target, and Smart Chart areas, but PDF report export and Smart Chart viewing are premium-only features.

This SPEC defines the next implementation lane. It is not an implemented-state report.

## Product Scope

The work has two connected deliveries:

1. Report output revision for every React-generated report.
2. Responsive React athlete portal with authentication, athlete tabs, and premium gating.

## Non-Negotiable Rules

1. Every generated report must fit on one A4 page.
2. Every printed/PDF report must use A4 paper with 2 cm margins on all sides.
3. Reports must use a premium visual model, suitable for athlete delivery.
4. Printed typography must use differentiated gray tones, not heavy saturated colors.
5. The app must be responsive for desktop, tablet, and mobile.
6. The athlete portal must have a login screen before athlete data is visible.
7. The athlete can see TAURUS and Target areas after login.
8. Only a premium subscriber can view Smart Chart.
9. Only a premium subscriber can export or emit PDF reports.
10. Smart Chart remains a visualizer. Do not duplicate TAURUS business logic in JSX.
11. TAURUS remains its own product lane and must keep its isolated data boundary.
12. In the responsive mobile/app-like athlete experience, `FREE` and `PREMIUM` athlete identity must come exclusively from the login session.

## Report Formatting Requirements

### Page Model

Target output:

```text
Paper: A4 portrait
Size: 210 mm x 297 mm
Margins: 2 cm top, right, bottom, left
Printable content area: 170 mm x 257 mm
Page count: 1 page per report
```

If the report content is too large, the system must compact, summarize, or move detail into a secondary on-screen view. It must not create a second PDF page without explicit future approval.

### Visual Standard

Reports must use a premium print-first style:

- white background
- no decorative gradients
- no dark blocks that waste ink
- strong spacing discipline
- compact cards or sections only when needed
- consistent title hierarchy
- muted divider lines
- typography in gray scale
- minimal accent color only if it still prints clearly in grayscale

### Gray Tone Typography

Recommended tone scale:

```text
Report title: #1f2933
Section title: #374151
Primary value: #111827
Body text: #4b5563
Secondary text: #6b7280
Muted label: #9ca3af
Divider: #d1d5db
Panel background, if used: #f7f7f8
```

Do not rely on red/green/blue color alone to communicate meaning. Any status must also use text, label, score, icon, or pattern that survives black-and-white printing.

### One-Page Compression Rules

Each report must prioritize:

1. athlete identity and session context
2. headline diagnosis
3. key metrics
4. chart or visual evidence
5. recommendation/prescription
6. coach note or next action

Long explanations, raw tables, audit trails, and debug details must stay outside the one-page PDF unless the user later approves an appendix format.

### Report Types To Audit

Audit every React-generated report surface before implementation closes:

- TAURUS Target output report
- Humanoide report
- Cartoes Coloridos report
- Duelo 20 report
- Smart Chart report or summary
- HCI summary/report surfaces if they emit printable output
- Training plan or prescription reports if PDF/export exists
- Admin-generated athlete delivery report, if present

The audit must produce a checklist with current file, report type, page status, and required formatting action.

## PDF Export Rules

PDF export is premium-only.

Free or non-premium athlete:

- can view allowed on-screen tabs
- cannot emit PDF
- sees locked premium action where PDF export would be
- must not receive a hidden PDF download path

Premium athlete:

- can emit PDF from approved report surfaces
- receives a one-page A4 PDF
- PDF matches the on-screen premium report preview
- PDF must be validated for one-page layout

Implementation must gate both UI and action handler. Hiding the button alone is not enough.

## Athlete Portal Scope

### Required Screens

Create or formalize these React screens:

```text
Login
Athlete Shell
TAURUS Tab
Target Tab
Smart Chart Tab
Premium Locked State
Report Preview / PDF Action
```

### Login Screen

The login screen must:

- appear before athlete data is visible
- identify the product clearly
- accept athlete credentials or local demo credentials depending on current backend readiness
- route to the athlete portal after successful login
- preserve responsive layout on mobile

If real authentication is not ready, implement a clearly isolated local/demo auth adapter so it can be replaced without rewriting the portal.

### Athlete Shell

The athlete shell must:

- show athlete identity after login
- expose tabs for TAURUS, Target, and Smart Chart
- preserve current TAURUS data path
- avoid mixing Admin controls into the athlete view
- keep navigation compact on mobile

### Mobile / App-Like Identity Rule

For the responsive mobile/app-like experience:

- `FREE` and `PREMIUM` athletes must not choose another athlete through a selector.
- Every athlete-specific screen, training recording flow, report preview, Smart Chart view, and Library recording flow must use the athlete identity from the login session.
- The logged-in athlete name is the source of truth for saving sessions, filtering summaries, and showing accessible data.
- Athlete selectors are allowed only for `ADMIN` workflows.
- If an Admin is using the responsive layout, the Admin selector may choose an athlete; this exception does not apply to `FREE` or `PREMIUM`.

Suggested visible tabs:

```text
TAURUS | Target | Smart Chart
```

### TAURUS Tab

Allowed for logged-in athletes.

Purpose:

- show the athlete's TAURUS context
- show approved or athlete-visible TAURUS outputs
- avoid Admin-only controls

### Target Tab

Allowed for logged-in athletes.

Purpose:

- show target session outputs available to the athlete
- preserve real target data
- avoid old HCI Target mock behavior unless explicitly rebuilt as real TAURUS

### Smart Chart Tab

Premium-only.

Free or non-premium athlete:

- tab may be visible but locked, or hidden depending on final UX decision
- locked state must explain that Smart Chart requires premium
- no Smart Chart data/model should be rendered behind the lock

Premium athlete:

- can open Smart Chart
- Smart Chart consumes approved sessions only
- Smart Chart remains fed by `buildTaurusSmartChartModel()` and related engines

## Subscription Contract

Add or standardize a subscription field in the athlete/session access model.

Recommended internal values:

```text
FREE
PREMIUM
ADMIN
```

Recommended derived permission model:

```js
{
  canViewTaurus: true,
  canViewTarget: true,
  canViewSmartChart: subscriptionTier === "PREMIUM" || role === "ADMIN",
  canExportPdf: subscriptionTier === "PREMIUM" || role === "ADMIN"
}
```

Canonical internal keys should remain in English. Portuguese remains the UI translation layer.

## Premium Locked State

Locked screens/actions must be polished, not broken empty states.

Required content:

- feature name
- premium requirement
- no fake metrics
- no preview that exposes restricted data

Example visible labels:

```text
Smart Chart Premium
Exportar PDF Premium
```

## Technical Boundaries

Do not break these existing boundaries:

- `TAURUS_TARGET_DB_V1` remains isolated.
- Smart Chart model generation stays in `src/services/taurusSmartChartEngine.js`.
- Target-specific intelligence stays in TAURUS target intelligence services.
- React components render models and UI states; they must not become calculation engines.
- Admin approval remains the gate for sessions used by Smart Chart.
- Do not revive old HCI Target mock behavior as production Target.
